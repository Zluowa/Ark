# OmniAgent 桌面端 A / B 完整交付稿

日期：2026-03-04  
版本：v1.0  
范围：A = 6 态高保真线框；B = 8 个关键动效逐帧脚本

---

## 0. 设计判断

OmniAgent 不该做成一个会不断拉走用户注意力的 AI 世界，而应做成一个安静、可控、结果导向的执行入口。
这份 A / B 方案把“舒服、可控、不抢注意力”落实到了页面与动画的每个细节：

- 每个 Surface 只回答一个问题。
- 每个状态只有一个主焦点。
- 动效只解释状态，不制造停留。
- 完成后必须快速落定并允许退场。

## 1. 外部基线

- **R1 Apple Human Interface Guidelines - Motion**：动效应传达状态、反馈与指导，而不是单纯装饰。  (https://developer.apple.com/design/human-interface-guidelines/motion)
- **R2 Apple Reduced Motion evaluation criteria**：应避免深度模拟、多轴运动、持续自动播放等容易引发不适的动画，并在必要时改用 dissolve、highlight、color shift。  (https://developer.apple.com/help/app-store-connect/manage-app-accessibility/reduced-motion-evaluation-criteria/)
- **R3 Apple WWDC23 - Meet ActivityKit**：Live Activities / Dynamic Island 的关键信息应保持 glanceable；只在离散用户动作后启动；最小态只放最关键内容。  (https://developer.apple.com/videos/play/wwdc2023/10184/)
- **R4 OpenAI Apps SDK - UI guidelines**：交互面应保持简单可预测；卡片限制 CTA 数量，不在单个卡片里塞深层导航或重复输入。  (https://developers.openai.com/apps-sdk/concepts/ui-guidelines)
- **R5 OpenAI - A practical guide to building agents**：系统设计应先从单 agent 与清晰工具开始，再按效果决定是否升级到更多编排；在满足目标的前提下优先优化成本与延迟。  (https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/)
- **R6 Anthropic - Writing effective tools for AI agents**：工具应边界清晰、目的明确；过多或重叠工具会分散 agent 的选择能力。  (https://www.anthropic.com/engineering/writing-tools-for-agents)

## 2. 状态机

Idle -> Compose -> Run -> Delivery -> Idle 为主路径；  
拖拽主路径为 Idle -> Intake -> Run -> Delivery；  
长任务主路径为 Run -> Stack -> Delivery / Idle；  
任一时刻都必须存在一个可退场动作：Esc、关闭、后台继续、回到 Idle。

## 3. A - 6 态高保真线框

### 01 Idle Island

- 用户问题：它在不在，我能不能马上开始？
- 默认尺寸：默认 96 x 34；Hover 112 x 38；运行中 128 x 38
- 进入条件：应用启动、完成任务后回退、用户唤出后无输入 2 秒回落。
- 退出条件：点击进入 Compose；拖拽进入 Intake；已有进行中任务时进入 Run / Stack。
- 主 CTA：点击或拖拽
- 注意力规则：只允许一个低频呼吸；提示文案不超过 8 个汉字；没有持续跳动。

**组件清单**
- 品牌胶囊 / 核心图形
- 状态点：空闲、运行、完成、异常
- 最近一次结果的轻提示（可选）
- 热键或拖拽提示（仅 Hover 显示）

**文案规则**
- 文案必须是任务型而不是聊天型，例如“拖文件进来”“说一句，交给我”。
- 如果显示最近结果，只说结果，不说过程，例如“PDF 已就绪”。

**边界与异常**
- 若最近任务失败，仅显示一个异常点和“可重试”，不展开长错误。
- 若某连接已过期，只在 Hover 时显示“飞书授权已过期”，不抢焦点。

### 02 Compose Sheet

- 用户问题：我怎么最快把任务说清楚？
- 默认尺寸：520 x 388
- 进入条件：点击 Idle、全局热键、从 Delivery 继续下一步。
- 退出条件：Enter 进入 Run；拖入文件进入 Intake；Esc 回到 Idle。
- 主 CTA：Enter 直接执行
- 注意力规则：输入框是唯一视觉中心；建议动作最多 3 个；无左侧导航、无工具首页。

**组件清单**
- 顶部上下文 chips：最近文件、已连接服务、当前会话
- 主输入框：单行起步，自动增高
- 3 个上下文相关建议动作
- 附件、发送、可选语音入口

**文案规则**
- 输入框 placeholder 必须写任务，不写“和我聊聊”。
- 建议动作优先显示高频碎事，例如“PDF 转图片”“下载字幕”“14:00 会议提醒”。

**边界与异常**
- 空输入时发送按钮可用，但 Enter 不触发，避免误操作。
- 若用户提到未授权服务，输入框下方只出现一个授权 chip，不弹整页流程。

### 03 Intake State

- 用户问题：我拖进来的文件被接住了吗？
- 默认尺寸：560 x 430
- 进入条件：拖拽文件进入岛的磁吸区域；或从 Compose 点击附件。
- 退出条件：点击开始进入 Run；Esc 回 Compose；不支持的文件进入错误卡。
- 主 CTA：开始
- 注意力规则：文件预览必须是锚点；推荐动作不超过 3 个；参数入口默认折叠。

**组件清单**
- 文件预览与元信息
- 自动识别的文件类型与推荐动作
- “换个说法”输入框
- 轻参数入口：输出格式、页码范围、分辨率等

**文案规则**
- 推荐动作使用强动词：压缩、提取、转图片、转音频。
- 参数标签必须简短，不出现工具名或底层服务名。

**边界与异常**
- 多文件拖拽时先按批次聚合成一个文件组卡片，不把 10 个 chip 一口气摊开。
- 若受版权或权限限制，替换主 CTA 为“查看限制说明”。

### 04 Run Sheet

- 用户问题：它是不是在干活，做到哪了？
- 默认尺寸：520 x 352
- 进入条件：任务被接受后立即进入。
- 退出条件：完成进入 Delivery；后台继续进入 Stack；失败返回 Compose 并带恢复建议。
- 主 CTA：后台继续（长任务时）
- 注意力规则：只强调当前阶段；无技术日志滚屏；重复动画上限 3 个循环。

**组件清单**
- 4 阶段阶段线：理解任务 / 准备能力 / 执行 / 交付
- 当前动作卡：正在做什么、是否可预测 ETA
- 后台继续 / 停止 / 诊断入口（默认折叠）

**文案规则**
- 文案必须指向真实动作，例如“正在把 PDF 渲染为长图”。
- 只有 ETA 可信时才展示；否则用“处理中”即可。

**边界与异常**
- 超过 8 秒自动出现“后台继续”，但不强制切走当前任务。
- 失败时只给 1 个主恢复建议和 1 个替代方案，不倒一堆堆栈信息。

### 05 Delivery Sheet

- 用户问题：结果在哪，下一步怎么拿走？
- 默认尺寸：540 x 388
- 进入条件：任务完成、结果文件或文本已经可交付。
- 退出条件：主 CTA 完成后回 Idle 或 Stack；若 8 秒无操作，结果保留在历史里并自然退场。
- 主 CTA：下载 / 复制 / 发送（依结果类型而定）
- 注意力规则：结果卡 420ms 内完全静止；下一步建议最多 3 个；无空白成功页。

**组件清单**
- 结果卡：文件名、类型、大小、生成时间
- 主 CTA + 最多 2 个次级操作
- 最多 3 个“下一步”建议

**文案规则**
- 用结果语言，不用庆祝语言，例如“结果已就绪”“zip 已生成”。
- 如果可发送到飞书 / Notion，只作为次级操作，不抢主 CTA。

**边界与异常**
- 多结果任务要先给一个“主结果”，其余结果折叠在展开列表中。
- 部分失败时必须写清楚什么成功、什么失败、是否还能先拿走已完成部分。

### 06 Stack View

- 用户问题：我现在同时有几件事？
- 默认尺寸：260 x 404
- 进入条件：长任务后台继续、第二个任务到来、Delivery 退场但仍保留产物。
- 退出条件：点击任一任务回到对应 Surface；全部完成后回 Idle。
- 主 CTA：打开主任务
- 注意力规则：永远只有一个主任务；可见任务最多 3 个；不做看板式复杂编排。

**组件清单**
- 1 张主任务卡
- 2 张折叠摘要卡
- 状态徽章与快速打开按钮

**文案规则**
- 标题优先写“结果 / 任务对象”，不要写内部流程名。
- 折叠摘要必须短：14 个汉字内。

**边界与异常**
- 超过 3 个任务时显示计数与历史入口，而不是继续堆栈。
- 若某任务需要用户确认，摘要卡显示一个醒目的待处理 badge。


## 4. B - 动效系统

### 4.1 Motion Token

- **T1 120ms**：点击反馈、复制确认、Hover 点亮
- **T2 180ms**：chip 进入、阶段切换、接住文件
- **T3 280ms**：Surface 展开 / 收起、Stack 交接
- **T4 420ms**：结果落定、完成退场

### 4.2 Reduce Motion 映射

- Hover glow + scale -> 亮度提升 + 提示淡入
- 磁吸弧线 / 速度暗示 -> 文件 chip 吸附 + 面板淡入
- 深度缩放 / 纵深位移 -> 交叉渐变 + 阴影层级变化
- 扫光 / shimmer 循环 -> 高亮点切换或直接静态更新
- 自动持续背景动效 -> 全部停用，改为静态状态提示

### 4.3 8 个逐帧脚本

#### 01 Hover Warmup

- 触发：鼠标进入 Idle Island 外围 24px 热区。
- 目的：告诉用户“我注意到你了”，但不把他拖入一个更重的交互。
- 默认时长：120ms 高亮 + 180ms 提示进入
**逐帧**
- 0ms：胶囊静止，无额外 halo。
- 120ms：外描边亮度提高约 6%，整体扩张 2-4px。
- 180ms：任务型提示从下方淡入，Y 轴 4px -> 0。
- 结束：所有运动停止，保留可读状态。
**Reduce Motion**：保留亮度提高与提示淡入，去掉尺寸变化、残影和光晕扩散。
**工程建议**：用 pointer proximity 触发；pointer leave 立即取消，不留悬挂状态。

#### 02 Compose Bloom

- 触发：点击 Idle Island 或按下全局热键。
- 目的：让输入面板像从岛里长出来，而不是突然弹窗。
- 默认时长：280ms Surface 变形
**逐帧**
- 0ms：胶囊是唯一焦点。
- 80ms：宽度先拉伸到即将展开的面板锚点。
- 160ms：面板从胶囊位置扩展，背景 blur 16 -> 0。
- 280ms：输入框获得焦点，光标出现。
**Reduce Motion**：胶囊与面板做交叉渐变，输入框直接聚焦，不做拉伸与深度变化。
**工程建议**：建议使用 matched geometry 或等效几何过渡；输入焦点晚于视觉展开 40-60ms。

#### 03 Drag to Capture

- 触发：文件进入磁吸区并松手。
- 目的：让用户感到文件被接住，而不是被扔进一个上传控件。
- 默认时长：180ms 接住 + 280ms 展开
**逐帧**
- 接近：Idle 出现轻磁吸暗示，文件速度视觉上放缓。
- 吸附：文件缩成一个 chip，落在 Intake 面板的起始位置。
- 展开：面板从岛向下生长，预览卡先出现。
- 稳定：推荐动作依次出现，等待用户确认。
**Reduce Motion**：去掉磁吸弧线和速度暗示，仅保留 chip 吸附与面板淡入。
**工程建议**：dropEntered 与 dropExited 状态必须可中断；文件 chip 是全流程的空间锚点。

#### 04 Suggestion Cascade

- 触发：上下文分析完成，系统要呈现 1-3 个建议动作。
- 目的：轻量补充而非突然塞满信息。
- 默认时长：3 x 120ms 顺序进入
**逐帧**
- 第 1 个：最相关动作先出现。
- 第 2 个：在 120ms 后淡入。
- 第 3 个：在 240ms 后淡入。
- 完成：所有 chip 停止运动。
**Reduce Motion**：三个 chip 同时淡入，不做错位位移。
**工程建议**：如果只有 1 个建议，不要补满 3 个占位；顺序由置信度决定。

#### 05 Phase Advance

- 触发：Run Sheet 的阶段切换。
- 目的：用最少的动效告诉用户系统还在工作。
- 默认时长：每次切换 180ms
**逐帧**
- 开始：当前阶段点亮，文本权重增加。
- 推进：前一阶段收敛成“已完成”状态。
- 切换：当前阶段切到下一阶段时只做一次扫光。
- 稳定：进入静止状态，等待下一次真实进展。
**Reduce Motion**：阶段点与标签直接切换高亮，不做扫光与横向运动。
**工程建议**：禁止无意义 shimmer 无限循环；没有真实进展就不更新阶段。

#### 06 Delivery Settle

- 触发：结果就绪，Run 切换到 Delivery。
- 目的：让“完成”有重量，但在 420ms 内落定，不继续兴奋。
- 默认时长：420ms 落定动画
**逐帧**
- 运行态：当前动作卡还在视图中心。
- 结果出现：结果卡从当前动作卡位置浮起 12-14px 并淡入。
- CTA 落定：主按钮从 0.98 缩放回 1.0。
- 完成：所有对象静止，进入可操作状态。
**Reduce Motion**：结果卡淡入，CTA 仅做亮度变化。
**工程建议**：主 CTA 的落定不应超过一次；避免 bounce 回弹。

#### 07 Micro Confirm

- 触发：复制、下载、发送等微操作完成。
- 目的：只给一次明确确认，不叠 toast。
- 默认时长：120ms
**逐帧**
- 按下：按钮产生 1px 深度反馈。
- 反馈：图标或文字切换到“已复制 / 已发送”。
- 确认：高亮保留 600ms 内，然后静止。
- 结束：恢复默认按钮态。
**Reduce Motion**：直接文字切换 + 高亮 1 次，不做缩放。
**工程建议**：优先做按钮内确认，避免额外 toast 覆盖 Delivery 的主信息。

#### 08 Stack Handoff

- 触发：长任务转后台，或第二个任务接管前台。
- 目的：在不中断当前思路的前提下，让用户知道“前台是谁，后台是谁”。
- 默认时长：280ms
**逐帧**
- 前台 A：当前 Sheet 在顶层。
- A 入栈：当前 Sheet 压缩成一张前景卡。
- B 接管：新的 Compose / Run 成为唯一主焦点。
- 稳定：A 留在 Stack 中持续更新摘要。
**Reduce Motion**：改为尺寸与阴影层级变化，不做纵深位移。
**工程建议**：Stack 的顺序由最近交互时间决定，而不是完成时间。

## 5. 前端落地建议

- 桌面壳建议用 SwiftUI + AppKit，同一组状态驱动 Idle / Compose / Intake / Run / Delivery / Stack。
- Surface 切换优先使用 source-target continuity（如 matched geometry），不要“从空中弹出”。
- 读取系统 Reduce Motion 设定，默认跟随系统，同时保留应用内细粒度开关。
- Delivery 的主 CTA 必须上下文驱动；复制、下载、发送三者只允许一个主按钮。
- 执行层优先走 workflow / tool runtime，避免把确定性任务交给长推理。
- 埋点优先关注：Compose 打开时延、拖拽接住率、阶段切换可信度、Delivery 主 CTA 使用率、Reduce Motion 打开率。
