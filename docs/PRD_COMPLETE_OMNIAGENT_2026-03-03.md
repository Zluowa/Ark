# OmniAgent 完整 PRD（灵动岛主线版）

版本：v1.0（完整稿）  
日期：2026-03-03  
状态：可执行  
适用阶段：2026 Q2-Q3  
文档目标：统一 OmniAgent 的产品内核、体验路线和商业落地优先级

---

## 0. 摘要

OmniAgent 的目标不是做“更会聊天的 AI”，而是做“最快把任务交付出来的 AI 工具系统”。

产品由两层构成：

1. B 端：给各类 Agent 提供统一工具执行与连接能力（One API Key）。
2. C 端：给用户一个极简入口（灵动岛），把复杂任务变成一句话和一次拖拽。

核心标准是四个字：`多快好省`。

---

## 1. 背景与机会

### 1.1 现状问题

当前 Agent 产品普遍存在三个问题：

1. 任务执行链路过长，用户等待时间高。  
2. 大量重复“写代码调用工具”的过程，token 消耗高。  
3. 前台体验复杂，用户只想要结果却被迫理解流程。

### 1.2 市场机会

1. 对开发者：需要“一次接入，多工具执行”的底层能力。  
2. 对终端用户：需要“轻入口 + 快交付”的工具型体验，不需要陪聊。  
3. 对企业：需要连接国内外服务、合规可控、可计量的执行层。

### 1.3 产品机会窗口

以 2026 年 AI 应用竞争阶段看，真正稀缺的不是“会说”，而是“会做并稳定交付”。

---

## 2. 产品愿景与定位

### 2.1 愿景

成为“所有 Agent 的工具执行基础设施”，同时提供“用户可直接使用的工具型 Agent 产品”。

### 2.2 一句话定位

OmniAgent = 所有 Agent 的工具执行层 + 面向用户的轻量交付入口（灵动岛）。

### 2.3 价值主张

1. 结果快：尽量秒级响应常见任务。  
2. 成本省：尽量减少不必要 token 与重复推理。  
3. 体验轻：入口清晰、交互简单、反馈明确。  
4. 能力多：工具覆盖 + 连接覆盖持续扩张。

---

## 3. 核心思想（产品内核）

### 3.1 方法论

1. 不造轮子：优先调用成熟工具与服务能力。  
2. 不空对话：优先交付结果，不优先解释。  
3. 不暴露复杂度：复杂逻辑留在后台，前台只显示任务状态和结果。  
4. 不增加负担：用户操作应接近“点击即聊、拖拽即处理”。

### 3.2 设计哲学

1. 小而美：界面精简但有品质。  
2. 少即是多：默认只保留高频动作。  
3. 状态可感知：每一步状态对用户都透明。  
4. 一切围绕交付：能下载、能发送、能复用，才算完成。

---

## 4. 用户与场景

### 4.1 目标用户分层

| 用户类型 | 典型角色 | 核心诉求 |
|---|---|---|
| B1 开发者 | Agent 开发者、MCP 作者 | 快速获得工具执行能力 |
| B2 团队/企业 | 运营、自动化团队 | 稳定连接服务并规模化执行 |
| C1 个人效率用户 | 内容创作者、办公用户 | 一句话完成日常工具任务 |
| C2 轻专业用户 | 媒体处理、文档处理用户 | 快速处理文件并拿到结果 |

### 4.2 核心场景

1. 文件处理：拖入 PDF/视频/图片，直接下达转换或压缩命令。  
2. 工具执行：输入“生成 UUID / 哈希 / 转码”等立即得到结果。  
3. 服务连接动作：如消息发送、文档写入、文件分发。  
4. 连续任务：在同一入口连续完成多个任务，不切换复杂页面。

### 4.3 关键 Jobs to Be Done

1. 当我有一个待处理文件时，我希望“拖进去 + 说一句”，立刻拿到结果。  
2. 当我需要跨服务执行动作时，我希望一次连接后重复可用。  
3. 当我处理多个任务时，我希望一直在同一个入口完成，不迷路。

## 5. 产品范围与边界

### 5.1 本期范围（In Scope）

1. 灵动岛入口（桌面）与常驻聊天入口。  
2. 文件拖拽任务闭环。  
3. 常用工具执行闭环。  
4. 连接中心（授权、状态、可用动作）。  
5. 执行记录与结果可追溯。  
6. 基础商业能力：用量、配额、权限、账单事件。

### 5.2 非本期范围（Out of Scope）

1. 复杂自治 Agent 编排可视化（高级工作流编辑器可后置）。  
2. 面向开发者的深度 IDE 类能力。  
3. 过多低频个性化 UI 选项。  
4. 用“聊天陪伴”替代“任务交付”。

---

## 6. 产品形态与信息架构

### 6.1 双端形态

1. C 端入口：桌面灵动岛 + 展开面板。  
2. B 端后台：Web 管理台（Tools / Connections / Usage / Settings）。

### 6.2 体验主路径（C 端）

1. 触达：看到灵动岛。  
2. 进入：点击即进入常驻聊天入口。  
3. 输入：文本指令或文件拖拽。  
4. 执行：状态反馈（等待/处理中/完成/失败）。  
5. 交付：可下载文件、可复制结果、可继续下一任务。

### 6.3 管理路径（B 端）

1. 连接服务并授权。  
2. 获取并管理 API Key。  
3. 查看执行记录、用量与配额。  
4. 配置权限策略与基础治理。

## 7. 功能需求（完整）

## 7.1 灵动岛入口系统

### 目标

建立“低负担、高确定性”的统一任务入口。

### 需求

1. 默认入口为聊天处理，不先跳工具页。  
2. 支持单击进入、拖拽进入、状态回看。  
3. 保持形态分级：紧凑态、展开态、多任务态。  
4. 支持关键任务动效反馈（拖入、处理中、完成、失败）。  
5. 展开与收起行为稳定可预期。

### 验收标准

1. 用户首次操作 10 秒内可开始任务。  
2. 拖拽文件后 1 次输入即可发起执行。  
3. 任意状态都能回到主入口，不会“卡死”。

## 7.2 聊天与指令系统

### 目标

让用户以最自然方式表达任务，系统负责转成执行。

### 需求

1. 输入框以任务导向文案呈现。  
2. 历史记录以“执行结果”为主，不以闲聊为主。  
3. 失败时给出明确下一步（重试、改参数、替代方案）。  
4. 支持同会话连续执行多任务。

### 验收标准

1. 用户能理解“现在在做什么”。  
2. 用户能理解“结果在哪、下一步怎么做”。

## 7.3 文件拖拽处理系统

### 目标

把“拖入文件 -> 指令 -> 结果”做成产品标志能力。

### 需求

1. 支持拖入主流文件类型（文档/图片/音视频）。  
2. 自动识别并推荐默认处理动作。  
3. 可修改指令后再执行。  
4. 处理成功后返回可下载结果。  
5. 支持进度和状态提示。

### 验收标准

1. 拖拽流程无需页面跳转。  
2. 成功任务有明确产物，失败任务有可执行建议。

## 7.4 工具执行系统

### 目标

让“工具能力”以最小学习成本被使用。

### 需求

1. 高频工具支持快速触发。  
2. 执行反馈统一：运行中、完成、错误。  
3. 结果格式统一：文本结果、文件结果、链接结果。  
4. 支持执行记录追溯。

### 验收标准

1. 用户不需要理解底层工具细节。  
2. 结果可复用、可二次操作。

## 7.5 连接中心（Connections）

### 目标

把复杂 OAuth/连接治理收敛到统一入口。

### 需求

1. 展示连接状态：可连接、连接中、已连接、异常。  
2. 支持连接与断开。  
3. 每个连接展示可执行动作范围。  
4. 提供连接健康与异常反馈。

### 验收标准

1. 用户可独立完成一次连接并验证。  
2. 连接异常可被快速定位与恢复。

## 7.6 执行记录与可追溯

### 目标

让每次执行可审计、可复现、可解释。

### 需求

1. 记录输入、执行时间、输出结果、状态。  
2. 支持按任务、时间、状态检索。  
3. 支持导出或分享必要执行信息（权限控制下）。

### 验收标准

1. 任意一次执行都能快速定位。  
2. 用户能理解历史记录并复用结果。

## 7.7 商业化基础能力

### 目标

支持从产品可用走向商业可运营。

### 需求

1. API Key 与租户隔离。  
2. 配额与并发限制。  
3. 用量统计与账单事件。  
4. 关键安全审计与合规基线。

### 验收标准

1. 不同租户数据与配额严格隔离。  
2. 用量和计费事件一致可核对。

---

## 8. 体验与交互规范（产品级）

### 8.1 交互原则

1. 入口一致：无论什么任务都从同一入口开始。  
2. 反馈即时：任何动作 200ms 内必须有反馈。  
3. 状态连贯：状态切换要有可感知过渡。  
4. 错误可恢复：错误不是终点，要给下一步。

### 8.2 文案原则

1. 面向行动，不面向解释。  
2. 减少术语，强调结果。  
3. 统一状态词典：等待、处理中、完成、失败。

### 8.3 动效原则

1. 动效表达状态，不做无意义装饰。  
2. 同类动作保持同类节奏。  
3. 长任务必须有进度感。

---

## 9. 核心指标（KPI）

### 9.1 产品效率指标

1. 首次任务完成时间（FTT）。  
2. 任务完成率（TCR）。  
3. 平均任务交付时长（TAT）。

### 9.2 体验指标

1. 同会话连续任务数。  
2. 失败恢复率（失败后继续完成）。  
3. 用户主观评分（快、顺、轻）。

### 9.3 商业指标

1. API 调用活跃租户数。  
2. 连接活跃率。  
3. 付费转化与留存。

---

## 10. 里程碑与发布计划

## M1（第 1-2 周）：核心闭环版

1. 打通“点击/拖拽 -> 指令 -> 结果”最短路径。  
2. 统一四态反馈。  
3. 完成关键失败兜底。

## M2（第 3-4 周）：体验一致版

1. 统一灵动岛与面板体验语言。  
2. 完成高频场景动效与反馈。  
3. 连接中心基本可用。

## M3（第 5-6 周）：商业就绪版

1. 完成用量、配额、权限、账单事件闭环。  
2. 建立可追溯与治理基线。  
3. 进入外部灰度验证。

## M4（第 7-12 周）：规模增长版

1. 扩展工具与连接覆盖。  
2. 完善稳定性与服务等级。  
3. 推进企业场景深度接入。

## 11. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 功能扩张过快导致入口变复杂 | 体验劣化 | 强制“一入口”原则，低频能力后置 |
| 动效与状态不一致 | 用户困惑 | 建立统一状态词典与动效规范 |
| 连接链路不稳定 | 执行失败率上升 | 增加连接健康检查与自动恢复机制 |
| 结果不可追溯 | 企业难落地 | 强化执行记录、审计与检索能力 |
| 过度技术导向偏离用户价值 | 商业转化受阻 | 所有需求按“交付结果价值”排序 |

---

## 12. 组织与协作要求

1. 产品、设计、工程以“任务交付效率”作为共同目标。  
2. 需求评审必须包含“是否增加用户负担”的审查项。  
3. 迭代验收必须包含真实任务端到端演示。  
4. 每周复盘必须输出“快了多少、完成率提升多少”。

---

## 13. 版本决策与取舍

### 必须坚持

1. 灵动岛入口优先于功能页扩展。  
2. 结果交付优先于长解释输出。  
3. 高频场景优先于低频长尾能力。

### 可以延后

1. 复杂编排 UI。  
2. 低频个性化设置。  
3. 非核心展示型功能。

---

## 14. 最终定义（用于对外对齐）

OmniAgent 的核心不是“一个会说话的系统”，而是“一个会交付结果的系统”。  
灵动岛不是装饰，而是统一入口。  
只要做到“点击即聊、拖拽即处理、结果即交付”，OmniAgent 就具备长期产品势能。
