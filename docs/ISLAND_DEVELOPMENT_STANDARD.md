# OmniAgent 灵动岛开发标准（V1）

## 1. 目标与边界

本标准用于统一 OmniAgent 灵动岛的产品结构、交互层级、动效语言与工具呈现方式。

必须遵循的核心原则：

1. 语言优先：默认入口是对话输入，不是工具列表。
2. 文件优先：文件进入后应直接进入任务理解与处理流程。
3. 结果结构化：完成态必须给出结果卡、主动作、后续建议。
4. 复杂度分层：先给最少可决策信息，细节按层展开。

禁止事项：

1. 不做“工具九宫格首页”作为主入口。
2. 不把用户暴露到 adapter/技术名层。
3. 不在执行态堆叠技术日志噪音。

---

## 2. 统一状态骨架

### 2.1 Idle Island（待机态）

规格：

1. 尺寸约 `88 x 32`（可按平台像素密度微调）。
2. 全黑主视觉，贴近 Apple Dynamic Island 语义。
3. 仅保留品牌点、轻状态、热键提示。

目标：

1. 降低信息负担，作为“可唤醒入口”而非“功能入口”。

### 2.2 Compose Sheet（轻输入态）

规格：

1. 宽度 `480-560`。
2. 仅包含：输入框、上下文 chips、`3-5` 个建议动作。

目标：

1. 让用户一句话直接发起任务。
2. 不展示工具筛选器与功能宫格。

### 2.3 Intake State（文件吸附态）

规格：

1. 文件拖入后出现吸附反馈。
2. 显示：文件 chip、类型识别、推荐动作、可编辑指令。

目标：

1. 把“上传完成”立即转化为“可执行任务”。

### 2.4 Run Sheet（执行态）

规格：

1. 使用阶段线，不展示实现细节日志。
2. 固定四阶段：
   - `理解任务 -> 准备能力 -> 执行 -> 生成结果`

目标：

1. 让用户理解系统正在推进，而不是等待黑箱。

### 2.5 Delivery Sheet（交付态）

规格：

1. 必须包含结果卡。
2. 必须包含主按钮：`下载 / 复制 / 发送`（按任务类型选择其一为主动作）。
3. 提供最多 `3` 个下一步建议。

目标：

1. 不允许只显示“已完成”一句话结束。

### 2.6 Stack View（多任务态）

规格：

1. 仅允许 `1` 个主任务高亮。
2. 仅保留 `1-2` 个折叠悬挂任务。

目标：

1. 提供并行感知，不演变为复杂流程看板。

## 3. 工具体系：任务化而非功能化

## 3.1 用户可见一级工具族（仅 6 个）

1. `Files`
2. `Knowledge`
3. `Web`
4. `Write`
5. `Connect`
6. `Ops`

说明：

1. 底层 adapter 可继续扩展，但不直接暴露给用户。

## 3.2 用户可见表达

应展示任务句式，例如：

1. 把 PDF 转图片
2. 从发票提取表格
3. 压缩视频
4. 发到 Notion
5. 生成 UUID

禁止直接暴露：

1. `pdf_extract_tables`
2. `notion_create_page`
3. `artifact_store_write`

---

## 4. 动效系统（必须落地）

## 4.1 五类核心动效

1. `Attract`：hover / drag over
2. `Intake`：文件吸附 / 上下文进入
3. `Transition`：`idle -> input -> run -> result`
4. `Progress`：长任务活性反馈
5. `Settle`：完成 / 复制 / 发送成功

## 4.2 统一时长档位

1. `120ms`：即时反馈
2. `180ms`：小组件切换
3. `280ms`：面板展开/收起
4. `420ms`：完成落定

## 4.3 工具族签名动效

1. `Files`：边角折叠、预览浮起
2. `Knowledge`：文本压缩、层级收拢
3. `Web`：多源点汇聚
4. `Connect`：路径发送
5. `Ops`：瞬时闪现 + 可复制反馈

标准要求：

1. 动效必须解释状态变化与层级关系，不能只做装饰。

## 4.4 可访问性（Reduce Motion）

当用户开启减少动态效果：

1. 平移/缩放降级为 `opacity / blur / color shift`。
2. 保留状态可理解性，不得出现“无反馈”。

---

## 5. 信息架构与文案规范

1. 默认先展示任务意图与结果，不展示技术参数。
2. 运行中显示“阶段 + 当前动作”。
3. 交付态使用“结果 + 主动作 + 下一步建议”模板。
4. 错误文案应给可执行修复建议，不只报错码。

---

## 6. 开发执行规范（DoD）

每个新增/改造任务在合并前必须满足：

1. 状态流完整：`Idle -> Compose -> Intake(可选) -> Run -> Delivery`。
2. 至少一个主动作可直接完成闭环（下载/复制/发送）。
3. 支持 Stack View 并发显示规则（1 主 + 1~2 折叠）。
4. 动效映射到五类系统，时长使用四档之一。
5. Reduce Motion 有降级路径。
6. 文案使用任务化表达，不暴露 adapter 名。

---

## 7. 当前版本实施优先级

P0（先做）：

1. `Files` 全链路（上传、识别、执行、交付下载）
2. `Compose/Run/Delivery` 三态打通
3. 常用任务签名动效（Files + Ops + Connect）

P1（随后）：

1. `Knowledge/Web/Write` 任务化入口
2. Stack View 任务并发体验优化

P2（精修）：

1. 全工具族签名动效统一
2. 视觉节奏与可访问性深度调优
